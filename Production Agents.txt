Module 1: What Agents Are (And What They Are Not)
An AI agent is not a smarter chatbot.
A chatbot is reactive. You ask, it answers. One pass, one response, conversation over.
An agent is autonomous. You give it a goal. It makes a plan. It uses tools to execute each step. It evaluates the results. It adjusts its approach. It keeps working until the goal is achieved or it determines the goal cannot be achieved.
The fundamental difference is the loop.
A chatbot's lifecycle: input → output. An agent's lifecycle: goal → plan → act → observe → think → act → observe → think → ... → result.
This loop — the agentic loop — is the foundation of everything in this course.
The loop works through the API like this. You send a request to Claude with a goal and a set of tools. Claude responds with either a final answer or a tool call. If it is a tool call, your code executes the tool and sends the result back. Claude processes the result and decides: call another tool, or finish. This repeats until Claude returns stop_reason: "end_turn".
The three mistakes that kill agents before they start:
Mistake one: checking for completion by parsing Claude's text. People look for phrases like "I'm done" or "here's my final answer" in the response. This is unreliable because natural language is ambiguous. The API gives you stop_reason: "end_turn" as a structured signal. Always use it.
Mistake two: setting arbitrary turn limits as the primary stopping mechanism. "Stop after 10 loops" either cuts off useful work or runs unnecessary iterations. Use turn limits as a safety net, not the primary mechanism.
Mistake three: dumping everything into one prompt. Agents need focused instructions. An agent trying to research, analyze, write, format, and deploy in one step will do all of them poorly. Chain the steps. One task per turn.
Module 2: Tool Design (The Most Important Skill You Will Learn)
An agent without tools is just a chatbot with extra steps.
Tools are what give agents the ability to interact with the real world. Search the web, query databases, read files, send emails, call APIs, process documents, execute code.
But here is the insight that most people miss: the quality of your tools determines the quality of your agent more than any other factor.
A poorly designed tool confuses the agent. A well-designed tool makes the agent reliable.
Every tool has four components:
Name. Clear, descriptive, unambiguous. search_product_database not helper_function_2. Claude reads the name first to decide if a tool is relevant.
Description. This is the most critical part. The description tells Claude WHEN to use the tool and WHEN NOT to use it. Vague descriptions lead to random tool selection. Precise descriptions lead to reliable behavior.
Bad description: "Useful for getting data." Good description: "Query the product database by product name, SKU, or category. Returns pricing, availability, and specifications. Use when the user asks about product details, stock levels, or pricing. Do NOT use for order status or shipping — use check_order_status for those queries."
Parameters. A strict JSON schema that defines what inputs the tool requires. Mark required fields as required. Add descriptions for each parameter. Do not make everything optional.
Return format. Always return a structured response with a status field (success, error, no_results) and the data. Never return None or empty responses — Claude will hallucinate to fill the gap.
Start with three tools maximum. Each additional tool is a decision Claude has to make. More tools means more chances for wrong decisions. Build three tools, get them working perfectly, then add more.
Module 3: Building Your First Agent (Step by Step)
Time to build. This is a complete, working agent.
The goal: Build a research agent that can search for information, perform calculations, and save findings to files.
The architecture:
A system prompt that defines the agent's role and behavior. Three tools: web search, calculator, note saver. The agentic loop that processes tool calls and feeds results back. Error handling for when tools fail. Logging to track what the agent does.
The system prompt:
markdown
You are a thorough research agent. When given a research question:

1. Break the question into sub-questions that need to be answered
2. Search for information on each sub-question
3. Cross-reference findings across multiple sources
4. Perform any calculations needed to support your analysis
5. Save key findings to a file for reference
6. Provide a comprehensive answer with supporting evidence

Rules:
- Always search for information rather than relying on memory for current data
- If a search returns insufficient results, try a different query
- Save important findings before moving to the next sub-question
- If you cannot find reliable information on a point, say so explicitly
- Cite your sources where possible
This prompt gives the agent a clear methodology. It knows the sequence of operations. It knows when to use each tool. It knows how to handle uncertainty.
Building the agentic loop:
The loop follows this pattern:
markdown
1. Send goal + tools to Claude
2. Receive response
3. If stop_reason == "end_turn": return final answer
4. If stop_reason == "tool_use": 
   a. Extract each tool call from the response
   b. Execute each tool
   c. Collect results
   d. Append assistant response + tool results to message history
   e. Send updated history back to Claude
   f. Go to step 2
5. Safety check: if turns > 20, force stop
This is the complete architecture. Everything else is implementation detail around this core pattern.
Error handling:
Every tool call should be wrapped in try/except. When a tool fails, return a structured error message that tells Claude what went wrong and suggests an alternative approach:
json
{
    "status": "error",
    "type": "timeout",
    "message": "Search timed out after 10 seconds.",
    "suggestion": "Try a simpler or more specific query."
}
Claude can then decide whether to retry, try a different approach, or acknowledge the limitation and move on. Without structured error messages, Claude either hallucinate a result or loops infinitely.
Module 4: Multi-Agent Systems
When a single agent needs to do too many things, performance degrades. The solution is to split the work across multiple specialized agents coordinated by a central orchestrator.
The architecture is hub-and-spoke:
A coordinator agent sits at the center. It receives the overall goal, decomposes it into subtasks, and delegates each subtask to a specialist agent.
Specialist agents are focused. A research agent only researches. A writing agent only writes. An analysis agent only analyzes. Each has a small, focused tool set optimized for its specialty.
All communication flows through the coordinator. Specialists never talk to each other directly. The coordinator passes information between them.
The critical rule that everyone violates:
Specialist agents do NOT inherit the coordinator's conversation history. Each specialist starts with a blank context. Everything it needs to know must be explicitly included in its prompt.
The coordinator knows everything. It has been building context across the entire conversation. When it spawns a specialist, most builders assume the specialist also knows everything. It does not. It knows nothing except what you put in its prompt.
python
# Wrong — assumes the specialist knows context
specialist_prompt = "Now analyze the data we discussed."

# Right — passes all required context explicitly
specialist_prompt = f"""
You are a data analysis specialist. Analyze the following dataset:

{json.dumps(dataset)}

Focus on:
1. Trends over the past 6 months
2. Outliers more than 2 standard deviations from the mean
3. Correlations between variables A and B

Prior findings from research phase:
{research_summary}

Return your analysis as a structured JSON object with keys:
trends, outliers, correlations, summary
"""
The explicit version is longer. It is also the only version that works.
Another common failure: The coordinator decomposes a topic too narrowly. If asked to research "the impact of AI on industries" and the coordinator only creates specialists for software and healthcare, it misses finance, education, manufacturing, media, and a dozen other industries.
Fix this by instructing the coordinator to enumerate the full scope before decomposing, and to validate coverage after decomposition.
Module 5: Production Deployment
The gap between "my agent works in a demo" and "my agent works in production" is enormous. Here is what you need to bridge it.
Authentication and authorization. If your agent accesses any external service, it needs proper authentication. API keys, OAuth tokens, and session management. Never hardcode credentials. Use environment variables.
Rate limiting. Both for external APIs your agent calls and for the Claude API itself. Build in delays between requests. Implement exponential backoff for retries. Track your token usage.
Logging. Log every message sent to Claude, every tool call, every tool result, every error, token usage per turn, total time per run, and the final output. Use structured JSON logging so you can query your logs programmatically.
Without logs, production debugging is impossible. When someone says "the agent did something weird," your logs are the only way to reconstruct what happened.
Monitoring and alerting. Set up alerts for: agent runs exceeding expected duration, error rates above threshold, token usage spikes, and tool failures. If your agent breaks at 2 AM, you want to know before your users do.
Graceful degradation. What happens when the search API is down? When the database is unreachable? When Claude's API returns an error? Each failure scenario needs a specific handler. The agent should never silently fail. It should either retry, fall back to an alternative, or clearly communicate the issue.
Cost management. AI agents can get expensive. Each turn consumes tokens. Each tool call adds processing time. A runaway agent can burn through your API budget in hours.
Implement token budgets per agent run. Track cost per task. Set hard limits that terminate the agent if costs exceed expectations. Review your cost data weekly and optimize the most expensive workflows.
Module 6: Testing and Evaluation
You cannot improve what you cannot measure.
Build a test suite. For each agent, create 20+ test cases that cover normal usage, edge cases, error conditions, and adversarial inputs.
For each test case, define:
The input (the goal or request)
The expected behavior (which tools should be used, in what order)
The expected output (what the final answer should contain)
Failure criteria (what would constitute a wrong answer)
Run your test suite after every change to the agent. Automate this. An agent that passes all tests today and breaks after a prompt change tomorrow needs to be caught immediately, not after users report problems.
Evaluate tool selection. Track which tools the agent selects for different types of queries. If the agent consistently picks the wrong tool, the tool description needs refinement. This is the most common source of agent quality issues and the easiest to fix.
Measure end-to-end quality. For research agents, have a human expert evaluate the accuracy and completeness of the agent's reports. For automation agents, compare the agent's output against what a human would produce. Score consistently and track improvement over time.
